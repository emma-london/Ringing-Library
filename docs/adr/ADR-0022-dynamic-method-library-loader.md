# ADR-0022: Dynamic method-library loader — live CCCBR fetch for power users

**Status:** Accepted
**Date:** 2026-07-11 (Proposed); Accepted 2026-07-11
**Deciders:** Emma (project owner)
**Related:** [ADR-0015](./ADR-0015-cccbr-method-library-data-source.md) (bundled snapshot vs live fetch — this ADR carves out an opt-in exception and so **partially amends** it), [ADR-0001](./ADR-0001-cross-platform-compute-architecture.md) (zero-I/O core — preserved), [ADR-0014](./ADR-0014-executor-offline-fallback.md) (graceful offline fallback), [ADR-0016](./ADR-0016-library-to-app-interface-contract.md) (public surface / SemVer / subpath exports), [ADR-0019](./ADR-0019-standard-set-subpath-export.md) (subpath-export precedent), `scripts/lib/parse-cccbr-text.mjs`, `src/method-library.ts`

## Context

A separate, shipped app built on `ringing-lib-ts` works well against the bundled
data (the `STANDARD_SET` subpath, ADR-0019 — the ~45-method local set). The next
want is a **power-user** capability: load the **full** CCCBR method library
(>25,000 methods) so a ringer can use a method that isn't in the bundled set —
including one added upstream recently.

This runs directly into **ADR-0015**, which considered exactly this data and
decided the full library ships as a **bundled static snapshot, *not* a live
fetch**, because a runtime fetch would reintroduce the network dependency the
zero-I/O core (ADR-0001) exists to avoid, for data that changes only ~monthly.

Two forces reopen that trade-off for the power-user case specifically:

- **Freshness.** The upstream data is regenerated roughly weekly (the file we
  fetched during this session was "Generated by Composition Library on Mon 06 Jul
  2026"). A power user reaching for the *full* library is often doing so
  *because* they want something new; a snapshot that has to be re-vendored and
  republished to update is the wrong tool for them. (The bundled snapshot remains
  exactly right for the common case — it is the always-available offline default.)
- **No mirror to maintain.** Emma's explicit preference: fetch **direct from
  CCCBR**, not from a project-hosted mirror, precisely so there is no second copy
  that can silently go stale or that someone has to keep running.

The other force — **don't duplicate this across apps** — is the standing project
position (ADR-0016: one published, versioned package; ADR-0019: shared capability
lives in the library on its own subpath, not re-implemented per app). So the
loader belongs in `ringing-lib-ts`.

The reconciliation with ADR-0015/ADR-0001 is that ADR-0015 rejected live fetch
**in the core**. It did not rule out an **opt-in loader shipped alongside the
core**. The core can stay zero-I/O and offline-guaranteed while a separate,
explicitly-opted-into module does I/O — the same quarantine ADR-0019 already uses
to keep bundled data off the root entry.

## Decision

Ship a **dynamic method-library loader** in `ringing-lib-ts`, as an **opt-in
module on its own subpath export** — **`ringing-lib-ts/cccbr-methods`** — separate
from the zero-I/O core. Apps that want live full-library access import it; apps
that don't never load it and the core's "no network I/O" guarantee is unchanged.

Concretely:

1. **Fetch direct from CCCBR, no mirror.** The loader fetches the CCCBR Text
   files over **HTTPS** —
   `https://methods.cccbr.org.uk/text/CCCBR_<Class><Stage>.html` — the same
   endpoints `scripts/refresh-method-library.mjs` uses at authoring time. (Note:
   the refresh script's `BASE_URL` is `http://`; HTTPS serves the identical
   content — verified this session — and must be used from a browser to avoid
   mixed-content blocking. The refresh script should switch to HTTPS too.)

   **Granularity — per (file-class, stage) is the primitive.** Each CCCBR file
   *is* one (file-class, stage) — `CCCBR_Surprise6.html` is Surprise Minor — so
   the headline call fetches exactly one file: one fetch, one parse, no
   over-fetching. This is the real access pattern (the motivating case: a ringer
   preparing a peal of one specific new method wants that one class at that one
   stage, and pulling the whole library would be waste). A **whole-stage
   convenience helper** is also shipped for the "all Minor methods" case — it
   loops the stage's class files and merges — but it is explicitly the heavier,
   opt-in path, never the default.

   The fetchable unit is CCCBR's **file-class** set (`Alliance`, `Delight`,
   `Hybrid`, `Plain`, `Principle`, `Surprise`, `Treble Bob`, `Treble Place` —
   the 8 in `CLASS_STAGES`), which is *not* the 10-value `MethodClassification`
   (the `Plain` file yields both `Bob` and `Place` methods; `Principle` yields
   Stedman/Erin/etc.). To let a power-user app build a valid picker and to let
   the loader avoid dead fetches, **ship the class×stage availability map**
   (from `CLASS_STAGES`) as part of this subpath. Jump/Dynamic stay excluded
   (extended notation the core can't model), exactly as at authoring time.

2. **Fetch raw Text and parse client-side** — no pre-parsed JSON, no build step,
   no hosted artifact. This reuses the existing pure parser. Because
   `scripts/lib/parse-cccbr-text.mjs` was written **pure and I/O-free** (raw text
   + (class, stage) → `MethodLibraryEntry[]`), it can be **promoted into shippable
   `src/`** (as TypeScript) and shared by both the loader and the refresh script,
   with no logic change. The power user accepts that some on-device processing
   happens — an explicit, opted-into cost, not a surprise on every app launch.

3. **Tag the loaded data with provenance** — a CCCBR upstream version *and* a
   fetch date. The loader returns not just a `MethodLibrary` but a small
   provenance record, e.g.:

   ```ts
   interface LibraryProvenance {
     source: 'cccbr-text';
     upstreamGenerated: string | null; // CCCBR "Generated by ... on <date>" marker, when present
     fetchedAt: string;                 // ISO timestamp of this fetch
     stages: number[];
     methodCount: number;
   }

   interface LoadedLibrary {
     library: MethodLibrary;
     provenance: LibraryProvenance;
   }
   ```

   `upstreamGenerated` is already extracted by the refresh script's
   `extractUpstreamDate` (same logic moves into the shared parser module). This
   is the runtime analogue of ADR-0015's `manifest.json` staleness marker. The
   **app** decides what to do with it: show "methods as of <date>", offer a
   refresh, or ignore it entirely. For many users this is one-and-done and never
   refreshed — the provenance record simply lets an app surface that choice
   rather than mandating a policy.

4. **The core is untouched.** `MethodLibrary` still takes a plain array and does
   no I/O; the loader is the thing that produces the array. This is the **first
   network I/O shipped in the package**, and keeping it on a dedicated subpath is
   what preserves the precise guarantee "the core (root entry) does no network
   I/O." Under ADR-0016 the new subpath is an additive, non-breaking surface
   addition (a MINOR).

5. **Failure degrades gracefully (ADR-0014 spirit).** A failed fetch (offline,
   CORS, upstream 5xx) must not strand the app: the loader surfaces a typed
   error/result, and the app falls back to the always-bundled `STANDARD_SET`.
   Consistent with ADR-0014's open "who owns fallback policy" question, the
   **loader provides the mechanism (it never silently swallows or hangs); the app
   owns the UX** (when to retry, whether to cache, what to tell the user).

**ADR-0015 is amended, not superseded:** its bundled-snapshot decision still
stands as the default and the offline guarantee. This ADR adds an *opt-in,
outside-the-core* live path on top, for the power-user full-library case.

## Resolved questions (2026-07-11)

All open questions were resolved with Emma the same day the ADR was drafted;
the ADR moved Proposed → Accepted on their resolution.

- **CORS — VERIFIED GREEN (2026-07-11).** Direct browser fetch only works if
  `methods.cccbr.org.uk` allows the cross-origin read. Tested from the deployed
  test bench's real origin (`https://emma-london.github.io`, HTTPS, a different
  origin — the true hostile case) via a `fetch()` in the page context:
  `GET https://methods.cccbr.org.uk/text/CCCBR_Principle5.html` and
  `…/CCCBR_Surprise8.html` both returned `status: 200`, response `type: "cors"`,
  and the body read successfully (44 KB and 2.0 MB respectively). A blocked
  request would have thrown `TypeError: Failed to fetch` with an opaque,
  unreadable response — this is the opposite, so CORS is open for the Text
  endpoints. (The `Access-Control-Allow-Origin` header itself reads back `null`
  from JS only because it isn't in the safelisted exposed set; the readable
  `cors`-type 200 is the definitive signal.) This clears the gate to
  implementation. *Note for the record:* CORS support is CCCBR's to change; it
  isn't a contractual guarantee, so the offline fallback to `STANDARD_SET`
  (below) remains load-bearing.
  - *Parse-cost observation:* the Surprise Major file alone is **2.0 MB** of
    Text — the client-side parse of a large stage is real work, reinforcing that
    this is an opted-into power-user action, not a launch-path cost, and a point
    in favour of per-class (not whole-stage-eagerly) fetch granularity below.
- **Granularity → per (file-class, stage) primitive + a whole-stage helper.**
  The single-file call is the headline (see Decision 1); a whole-stage
  convenience that loops-and-merges the stage's class files ships alongside it,
  flagged as the heavier opt-in path.
- **Caching → pure-fetch; the app owns the cache.** The loader fetches, parses,
  and returns the library + provenance, and does **not** persist anything. An
  app that wants offline reuse caches the result itself, keyed on
  `provenance.upstreamGenerated`. Rationale: smallest surface, honours ADR-0014's
  "app owns policy," and keeps the loader environment-agnostic (no IndexedDB
  assumption). Built-in caching can be added later additively without breaking
  the pure-fetch signature. Rejected **for v1** (not permanently): a built-in
  IndexedDB cache — more convenient, but a bigger, browser-only surface, and
  **IndexedDB does not exist in Node**, so baking it in would cost the loader its
  current environment-agnostic portability.

  - **Revisit scheduled — Phase 5, as a spike (2026-07-11).** Offline use is a
    real goal here, not a nicety: the consuming apps are often used on mobile in
    remote places with no network coverage, and should keep working with as much
    of the library as was last loaded. The open design question the spike must
    answer is how to add persistent caching **without breaking Node
    compatibility** — the likely shape is a **pluggable storage-adapter seam**
    (the loader defines a small async get/put interface; a browser app supplies
    an IndexedDB-backed implementation, Node/other environments supply their own
    or none) rather than hardcoding IndexedDB into the loader. The spike should
    measure the impact (stored size per stage, cold vs warm load times, real
    on-phone offline behaviour) and settle the invalidation policy
    (serve-cached-first; when, if ever, to re-check upstream). Its outcome lands
    as its own ADR or an amendment here.
- **Subpath name → `ringing-lib-ts/cccbr-methods`.** Names both the source
  (CCCBR) and the payload (methods), and is unambiguous at the import site that
  this is live CCCBR I/O, not bundled data. Preferred over `/cccbr` (less
  specific) and `/loader` (source-agnostic — hides that it's CCCBR-only today).
- **`MethodLibrary` merge ergonomics → app-side concat for now.** A power user
  typically wants the bundled `STANDARD_SET` *plus* one or more fetched
  (class, stage) files. `MethodLibrary` is already iterable, so an app composes
  with a spread — `new MethodLibrary([...STANDARD_SET, ...loaded.library])` — no
  new API needed. One caveat to document: a method present in both sources
  (e.g. Cambridge Surprise Minor is in the standard set *and* in the Surprise
  Minor file) yields a duplicate; **de-dupe by `id`** (both sources carry the
  CCCBR `id`) if that matters. A formal `merge`/dedupe helper is deferred until
  the pattern recurs.

## Options Considered

- **Keep ADR-0015 as-is (bundled snapshot only), no live path.** Rejected for
  the power-user case: updating requires re-vendoring and republishing the
  package, which cannot deliver a method added upstream this week. Fine for the
  common case, which is why the snapshot stays as the default.
- **Project-hosted mirror of pre-parsed JSON shards (fetched at runtime).**
  Rejected per Emma: a mirror is a second copy to keep fresh and a service to
  run; it can silently go stale. Direct-from-source avoids both. (Retained only
  as a *fallback* if CORS forecloses direct fetch — see Open Questions.)
- **Fetch pre-parsed JSON instead of raw Text.** Rejected: there is no
  authoritative JSON endpoint upstream, so this would require the mirror we just
  rejected. Raw Text + the existing pure parser needs no hosted artifact. The
  extra client-side parse cost is acceptable for an opted-in power-user action.
- **Put the fetch inside the core / `MethodLibrary`.** Rejected: violates
  ADR-0001's zero-I/O core and ADR-0015's core-stays-offline decision, and would
  make *every* consumer carry network code. A dedicated subpath keeps the core
  pure and the capability opt-in and tree-shakeable (the ADR-0019 pattern).
- **Re-implement the loader in each app.** Rejected per Emma and ADR-0016: it
  duplicates non-trivial fetch+parse+provenance logic and lets each copy drift
  (the ADR-0016 `callsFor()` drift is the cautionary precedent). Shared in the
  library, versioned once.
- **No provenance / no version tag.** Rejected per Emma: without an upstream
  marker + fetch date, an app can't tell the user how current their methods are
  or offer a sensible refresh. The markers are cheap (already parsed) and let the
  app — not the library — own the refresh policy.

## Consequences

- **New public surface:** the `ringing-lib-ts/cccbr-methods` subpath export,
  exposing the loader (single-file + whole-stage calls), the class×stage
  availability map, and its `LoadedLibrary`/`LibraryProvenance` types; additive
  and non-breaking under ADR-0016 (a MINOR). The root entry and the zero-I/O
  guarantee for it are unchanged.
- **The parser becomes shared, shipped code.** `parse-cccbr-text` moves from a
  `scripts/` `.mjs` into `src/` TypeScript, imported by both the loader and the
  refresh script (single source of truth). Its existing tests
  (`src/tests/parse-cccbr-text.test.ts`, real CCCBR rows) carry over and should
  be extended for the loader path.
- **First shipped network I/O in the package.** Documented and quarantined to
  the subpath; worth a note in `README`/`CONTRIBUTING` so the "core does no I/O"
  claim stays precise ("the *core* does no I/O; the opt-in `/cccbr` loader does").
- **A hard dependency on CCCBR's live endpoint and its CORS policy** for the
  live path — see Open Questions. The bundled `STANDARD_SET` remains the offline
  guarantee, so an outage degrades to "standard set only," never to a broken app.
- **Coupling to the CCCBR file layout** (per-class-per-stage URLs, Text columns)
  moves from authoring-time-only (the refresh script) to *also* runtime. If CCCBR
  changes URLs or format, the shipped loader breaks for end users, not just a
  maintainer re-running a script — a sharper failure mode to keep in mind.
- **Roadmap:** this is new scope beyond the current Phase 4a/4b plan — a
  power-user capability sitting on the method-data side rather than the engine
  side. Tracked as its own line item (named **Phase 4aa** — it extends Phase 4a's
  method-database work and sits outside the engine sequence 4b/5), not smuggled
  into 4b.

## Implementation (2026-07-11)

Built the same day as acceptance:

- **`src/cccbr/parse-text.ts`** — the CCCBR Text parser, promoted from
  `scripts/lib/parse-cccbr-text.mjs` into shipped, typed `src/` (plus
  `extractUpstreamDate` and the `CccbrFileClass` type). The old `.mjs` was
  deleted; `scripts/refresh-method-library.mjs` now imports the built copy
  (`../dist/cccbr/parse-text.js`) so `npm run data:refresh` builds first
  (mirroring `engine:live-diff`), and its `BASE_URL` moved to HTTPS.
- **`src/cccbr/index.ts`** — the loader: `loadMethods(fileClass, stage)`
  (single-file primitive), `loadStage(stage)` (whole-stage merge over the
  iterable `MethodLibrary`), the availability map (`CCCBR_AVAILABILITY`,
  `CCCBR_FILE_CLASSES`, `classesForStage`, `stagesForClass`, `isAvailable`), the
  `LibraryProvenance`/`LoadedLibrary` types, `CccbrLoadError`, and an injectable
  `FetchLike` (defaults to global `fetch`; lets a proxy or a test double slot in,
  and keeps the module free of a hard `@types/node` fetch dependency).
- **Wiring** — `package.json` gains the `./cccbr-methods` subpath in `exports`,
  version bumped to **1.3.0** (additive MINOR, ADR-0016); loader kept **off**
  `src/index.ts`. `CHANGELOG.md` updated.
- **Tests** — `src/tests/cccbr-methods.test.ts` (12 tests, injected fake fetch:
  URLs/stems, provenance, whole-stage merge, `RangeError` on an unpublished
  (class, stage) with no dead fetch, `CccbrLoadError` on network/HTTP failure);
  the ported parser's 8 tests still pass. **294 tests green, clean `tsc`.**
- **Not verified in-sandbox:** the live fetch — this environment has no outbound
  route to `methods.cccbr.org.uk` (the same limitation ADR-0015 hit for the
  refresh script). The *failure* path is confirmed (a real network error surfaces
  as `CccbrLoadError`), and the browser-origin CORS check (above) already
  confirmed the real endpoints are reachable and readable; the happy-path live
  fetch runs in a networked environment / the consuming app.
