# Architecture Decision Records

This directory holds the project's Architecture Decision Records (ADRs). Recording significant decisions here is the **default for the whole project** — see `../../CLAUDE.md`.

## What gets an ADR

Anything that shapes the public surface, the architecture, the phasing, the toolchain, or a trade-off a future contributor would otherwise have to reverse-engineer. When in doubt, write one — they are cheap and the rationale is the point.

Decisions still being worked out also get an ADR, in `Draft` status, framing the open questions so the thinking survives between sessions.

## Format

Each ADR is one Markdown file, numbered and titled: `ADR-NNNN-short-kebab-title.md`. Use the next free number; never reuse one.

A record carries, at minimum:

- **Header** — `Status`, `Date`, `Deciders`, and `Related` (links to other ADRs).
- **Context** — the forces in tension and what prompted the decision.
- **Decision** — what was chosen, stated plainly.
- **Options Considered** — the alternatives, with why each was or wasn't chosen. This is the part that prevents relitigation; do not skip it.
- **Consequences** — what becomes easier, what becomes harder, what to revisit.

`Status` moves `Draft` → `Proposed` → `Accepted`, or `Superseded by ADR-NNNN`. Never delete or silently rewrite a superseded ADR — mark it, link forward, and keep the history.

Full sub-lead calling (Option A) deferred as a future ADR-0006 successor.
- **ADR-0008** — *(Accepted)* A declarative corpus of known-true/false test fixtures. `src/tests/fixtures/known-touches.json` centralises composition-verdict oracles (a JSON file, not a TS module); calls are derived via `standardCalls` (ADR-0009), not stored per-fixture; `stedmanTriplesComposition` generalized to `stedmanComposition` (Triples upwards) so the corpus can hold higher-stage Stedman fixtures; `docs/example-touches.md` is now generated from the corpus. Method-level call-structure facts and search-space aggregate facts stay out of scope, left where they were.
- **ADR-0009** — *(Accepted)* Generic method & call construction. `standardCalls(method)` dispatches by name family — `Grandsire*` (Doubles upwards) and `Stedman*` (Triples upwards) to their special-case factories, `Stedman*` at stage 5 (Doubles) throws (deferred to Phase 5) — with a stage-independent bob-`14`/single-`1234` default for everything else. `MethodClassification` deliberately not the dispatch key.
- **ADR-0010** — *(Accepted)* Deploying the test bench as a public web app. A Vite SPA under `app/` that imports straight from `src/` (no hand-bundled global), published to GitHub Pages via GitHub Actions; `base: /Ringing-Library/`. Supersedes the self-contained `ringing-test-bench.html`.
- **ADR-0011** — *(Accepted; §5 Stedman exclusion superseded by ADR-0012)* A bounded composition searcher in the app, ahead of the Phase 4 engine. A capped, shortest-first, truth-pruned enumerator (`src/search.ts` / `searchTouches`, "Search" tab) for the lead-end call methods, with hard length/result/node ceilings and a stable interface Phase 4 swaps the implementation behind. Scoped *not* to be the throwaway searcher ADR-0003 forbids: naive body, no reachability DP/rank-bitset, results re-provable by `Touch.prove()` and live-diffed against the C++ `grandsire_solver` prototype. (Stedman was out of scope; ADR-0012 brings it in.) Expected to be superseded by the Phase 4 search engine.
- **ADR-0012** — *(Accepted)* Stedman / six-based calling in the bounded searcher. Supersedes ADR-0011's Stedman exclusion by adding `searchStedmanTouches` — a per-six iterative-deepening, truth-pruned DFS (deciding plain/bob/single at each six, so calls are single-character and the compound-symbol problem vanishes), same `SearchReport` interface and same hard ceilings. Verified by row-for-row match to `Touch(stedmanTriplesComposition(...))`, by re-proving every result, and against an independent truth-checked enumeration (bobs-only ≤84 = 46 touches, incl. plain course and SLQ). Search tab now covers Stedman Triples.
- **ADR-0013** — *(Accepted)* Phase 4 pre-work and split, superseding ADR-0003's Phase 4 phasing. Inserts **pre-4**: finalize and implement ADR-0008 then ADR-0009, in that order, while the work is still pure TS, plus (unordered relative to those two) extending `BELL_NAMES` beyond Maximus — same shape of work, rides along with the checkpoint. Splits Phase 4 into **4a** (WASM engine core — language spike, port the validated algorithms, live-diff vs the C++ prototypes; a checkpointable deliverable with no plumbing dependency) and **4b** (execution plumbing — `Executor` seam, budget, deterministic mode, resumable chunk protocol, streaming/progress/cancel — then swap it into `src/search.ts`, superseding ADR-0011/ADR-0012's bounded DFS bodies). *Addenda (2026-07-03):* 4a also scoped to vendor a bundled CCCBR method-library snapshot (ADR-0015); 4b also scoped to define `Remote`→`Worker`/`Local` graceful offline fallback (ADR-0014).
- **ADR-0014** — *(Draft)* Offline / degraded-connectivity behavior for the `Executor`. `Remote` (the one executor variant needing a network) must fail gracefully, falling back to `Worker`/`Local` rather than erroring outright. Frames open questions (detection, fallback target, resumability, UX visibility) for Phase 4b to resolve.
- **ADR-0015** — *(Draft)* CCCBR method library data source: bundled snapshot vs live fetch. Vendors the full CCCBR method library as a static snapshot (not a runtime fetch), since the upstream data changes only ~monthly and a live fetch would reintroduce a hidden network dependency the zero-I/O core was designed to avoid. Frames open questions (refresh mechanism, format, lazy-loading, staleness marker, relationship to `STANDARD_METHODS`) for Phase 4a to resolve.
- **ADR-0018** — *(Accepted; amends ADR-0017)* Generalize the engine core to any lead-head method. The first engine was Grandsire-Triples-specific and used no library types; this reworks it into a generic `LeadHeadEngine(method, calls)` over any treble-hunt lead-head method (Plain Bob, Grandsire, surprise, …), with `GrandsireTriplesEngine` kept as a thin preset (and the C++ live-diff path). Library types at the boundary (Method + CallDefinition in, tables built from `Change`, `Composition` out, re-provable by `Touch`); flat rank/bitset kernel stays internal (ADR-0002 `SearchTruth`). Principles (Stedman) stay out of scope, mirroring the `searchTouches`/`searchStedmanTouches` split. The Grandsire-specific singles-parity prune is dropped for a universally-correct reachability-only prune — which also fixed a latent C++ `find` bug that dropped snap touches. Validated against `searchTouches` + `Touch.prove()` across five methods; stage ceiling at Royal (10).
- **ADR-0017** — *(Accepted; amends ADR-0013)* Phase 4a engine language. A real spike (`engine/`) found AssemblyScript ~2.3× **slower** than optimized TS on the ranked-DFS + bitset-truth kernel (C++ native 97 ms → optimized TS 185 ms → AS-wasm 422 ms at `count(16)`, all matching the C++ oracle), boundary overhead a negligible 7.5 ns/call, and Rust/wasm-pack unprovisionable in this sandbox/CI (rustup allowlist-blocked, no sudo, apt rustc lacks the wasm32 target). Decision: build the 4a engine core in **optimized TS behind a stable seam** now; keep **Rust/wasm-pack** (not AssemblyScript) as the future WASM drop-in behind the same interface. Revises ADR-0013's "WASM module" deliverable wording only; the pre-4/4a/4b split and all other 4a scope stand.
- **ADR-0016** — *(Accepted)* Library-to-app interface contract: versioning, distribution, deprecation. The public surface is exactly `src/index.ts`'s exports; SemVer governs it (`0.x` may break on any MINOR — pin exact versions pre-1.0); future apps consume a **published, versioned package**, never shared source; the test bench (`app/`, `../src` import) is the **sole, named exception**, not a precedent; deprecated exports get JSDoc `@deprecated` + a runtime `console.warn` + a guaranteed one-MAJOR-cycle window before removal; `CompositionJSON` gets its own `schemaVersion` field since persisted/shared compositions are a compatibility surface independent of the function API. Prompted by `app/main.ts`'s `callsFor()` silently drifting out of sync with ADR-0009's `standardCalls` before being caught and fixed.
